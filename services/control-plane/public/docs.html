Cavix documentation

Cavix is the AI code reviewer that proves its findings before it speaks. It reproduces a suspected bug in an isolated sandbox, applies the fix, and re-runs your tests, so its comments are verified facts, not guesses. This guide covers everything you can do from the dashboard.

Sign in & connect GitHub

Click Continue with GitHub on the login page. Cavix asks GitHub to authorize it (the standard read:org + user:email consent), then brings you back. On the Repositories page you'll see every organization you belong to; pick one, and toggle Cavix on for any repo, all without leaving the site.

  • The first person to sign up for an organization becomes its owner.
  • New GitHub sign-ups start a 14-day trial so private repos work immediately.
  • The one-time GitHub App install is a single click via Install GitHub App (GitHub requires that consent screen).

Your first review

Once a repo is connected, open a pull request. Cavix runs its 13-stage pipeline and posts:

  • A PR summary comment (walkthrough, changed-files table, sequence diagram, review-effort estimate).
  • Inline findings with severity and, for verified bugs, the failing test that proves them.
  • A ✓/✗ status check that can gate merges.

@cavix commands

Comment on any PR (write access required):

@cavix review     # fresh full review (clears stale comments & cache)
@cavix summary    # regenerate the PR summary
@cavix resolve    # resolve Cavix's own threads
@cavix pause      # stop auto-reviews on this PR
@cavix resume     # re-enable auto-reviews
@cavix help       # list commands
@cavix <question> # free-text Q&A about the PR

PR summaries

Every review posts a summary comment to the PR: a plain-English walkthrough, a table of changed files with one-line descriptions, auto-generated labels, linked issues/tickets, an optional sequence diagram, and a 1–5 review-effort estimate. Regenerate any time with @cavix summary.

Review tone

Set how Cavix writes comments in Review settings → Tone:

  • Concise, short, to the point.
  • Detailed, thorough explanations and context.
  • Educational, teaches the "why" with references.
  • Assertive, direct and prescriptive.
  • Chill, friendly and low-pressure, nits downplayed.

.cavix.yaml config

Commit a .cavix.yaml (or .cavix.json) to your repo root to configure Cavix in code:

autoReview: true
reviewDraftPRs: false
tone: concise
pathFilters:
  include: ["src/**", "services/**"]
  exclude: ["**/*.min.js", "**/generated/**"]
agents:
  disabled: [standards]
preMergeChecks:
  enabled: true
  rules:
    - "Every new endpoint has an auth check"
    - "Public API changes require a changelog entry"
failOn: [critical]

Pre-merge checks

Pre-merge checks are an optional gate, off by default. An owner turns them on in Review settings → Pre-merge checks and writes rules in plain English. Each rule compiles into a deterministic, non-bypassable check that runs before merge; a failing rule fails the Cavix status check. Combine with branch protection to make it a hard gate, ideal for regulated teams.

BYOK & models

Bring your own key on AI & BYOK: pick a provider (Anthropic / OpenAI / Google / self-hosted) and model, and paste your key. It's encrypted at rest (AES-256-GCM) and only a fingerprint is ever shown. Cavix never marks up tokens, you pay your provider directly.

Self-host & air-gap

Cavix ships a Helm chart and Terraform. In air-gapped mode a deny-all-egress NetworkPolicy plus an application egress guard guarantee zero outbound calls; the model runs in your cluster. See the deployment guide for the full runbook.

Security

  • Zero-retention, customer code is destroyed after each review; only metadata is kept.
  • Encrypted secrets, BYOK keys and OAuth tokens are encrypted at rest.
  • SSO/SAML, SCIM, RBAC, tamper-evident audit log on enterprise.
  • Sandbox isolation, untrusted code runs network-less with strict caps; assume-breach, defense-in-depth.
